# 低代码平台架构治理与重构方案（企业级）

## 1. 当前架构问题清单

1. **模块耦合偏高**
   - 前端 `main.ts` 在不同阶段被切换为不同入口，说明运行时与设计时尚未解耦。
2. **重复代码存在**
   - 页面渲染与设计器渲染均有栅格/节点渲染逻辑。
3. **权限体系分散**
   - RBAC 设计存在文档与模板片段，但运行时统一权限网关缺失。
4. **Schema 不统一**
   - Form Schema 与 Page Schema 在不同目录定义，缺少统一版本协议。
5. **动态渲染引擎安全性不足**
   - 公式/条件表达式存在 `Function` 执行方式，缺少沙箱与白名单。
6. **组件注册体系割裂**
   - 表单注册器与页面设计器注册器未统一插件扩展协议。
7. **事件系统缺少标准事件总线**
   - 表单事件与页面事件模型不一致。
8. **数据源规范不统一**
   - 数据源接口返回契约、鉴权、缓存策略缺少统一标准。
9. **API 规范不统一**
   - 部分接口返回 `String`，部分返回对象，缺少统一 `ApiResponse` 包装。
10. **性能风险**
   - 运行态数据与规则多在内存结构，缺少缓存分层和索引策略。

## 2. 风险等级

- **P0（高）**：动态表达式执行安全风险、API 返回不统一导致前后端不稳定。
- **P1（中高）**：Schema 多版本协议未统一，导致后续演进和兼容成本高。
- **P1（中高）**：事件系统与组件注册系统分裂，插件化能力受限。
- **P2（中）**：渲染逻辑重复导致维护成本增加。
- **P2（中）**：权限与多租户能力偏文档化，缺少统一接入层。

## 3. 重构方案

### 3.1 核心原则
- 内核稳定、能力外置、协议先行、版本兼容。
- 先统一契约（Schema/API/Plugin SPI），再拆模块。

### 3.2 三阶段重构
1. **阶段A：协议统一**
   - 统一 `MetaSchemaEnvelope`、`ApiResponse`、`DataSourceResponse`。
2. **阶段B：引擎解耦**
   - Form/Page/Workflow/Rule 各自 runtime + shared core。
3. **阶段C：平台化治理**
   - 插件市场、多租户配额、灰度发布、审计中心。

## 4. 模块拆分建议

- `platform-core`：租户、鉴权、API 规范、审计。
- `meta-core`：Schema 协议、版本、发布、回滚。
- `form-runtime`：表单渲染、校验、事件、公式。
- `page-runtime`：页面渲染、布局、生命周期。
- `workflow-runtime`：流程执行、任务、状态机。
- `plugin-runtime`：组件扩展、事件扩展、数据源扩展。
- `design-studio`：设计器聚合层（前端）。

## 5. 插件化建议

- 定义统一插件清单 `plugin-manifest.json`：
  - `name/version/compatibility/extensionPoints/routes/menus/permissions`
- 扩展点标准化：
  - `component.register`
  - `event.handler`
  - `datasource.adapter`
  - `formula.function`

## 6. Schema 统一方案

统一顶层协议：

```json
{
  "schemaType": "FORM|PAGE|WORKFLOW",
  "schemaCode": "...",
  "version": "1.0.0",
  "meta": {"tenantId": "...", "status": "draft|published"},
  "payload": {}
}
```

统一收益：
- 兼容多引擎
- 支持发布快照
- 支持跨租户隔离

## 7. 目录重构方案

```text
backend
├── platform-core
├── meta-core
├── form-runtime
├── page-runtime
├── workflow-runtime
└── plugin-runtime

frontend/src
├── platform
│   ├── schema
│   ├── api
│   └── plugin
├── runtimes
│   ├── form
│   ├── page
│   └── workflow
└── studio
    ├── form-designer
    ├── page-designer
    └── workflow-designer
```

## 8. 核心代码重构（本次已落地）

1. 新增后端统一 API 响应 `ApiResponse`，为后续接口治理提供标准。
2. 新增后端统一 Schema 信封 `MetaSchemaEnvelope`，用于 FORM/PAGE/WORKFLOW 统一协议。
3. 新增前端统一 Schema 协议定义 `meta-schema.ts`。
4. 新增前端统一 API 客户端封装 `api-client.ts`，统一请求入口与错误处理。
